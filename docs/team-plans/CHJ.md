# 조현정 1차 배포 계획: 서비스 등록

## 담당 범위

- 서비스 페이지 진입과 카테고리별 서비스 조회
- 서비스 목록 2열 5행, 페이지당 10개 노출
- 첫 서비스 등록 조건 체크리스트 모달
- 서비스 등록 폼
- 서비스 썸네일 업로드 조건
- AI 도우미 추천
- 서비스 URL 검증과 Google Web Risk 검사
- 서비스 데이터가 피드백, 홍보, 신고, 마이페이지에 연결되도록 제공

## 작업 브랜치

- 기본 브랜치: `feature/service-registration`
- 필요 시 세부 브랜치: `feature/service-list`, `feature/service-registration-url-check`, `feature/service-ai-helper`

## 프론트엔드 작업 전제

- 피그마 MCP로 생성한 프론트 기초 코드는 `frontend` 브랜치에서 먼저 구현되고, 통합 관리자가 `develop`에 병합한다.
- 조현정 작업은 `frontend`가 병합된 `develop`에서 `feature/service-registration` 브랜치를 만든 뒤 시작한다.
- 서비스 페이지, 서비스 목록, 서비스 등록 폼, 체크리스트 모달은 피그마 기반 기초 코드를 최대한 유지한다.
- 피그마 의도와 다르게 생성된 서비스 화면 구조나 컴포넌트만 담당 범위 안에서 보정한다.
- 기능 연결 시 목업 데이터는 `GET /services`, `POST /services`, `GET /services/{service_id}` 등 실제 API 호출로 교체한다.

## 주요 계약

- 주요 테이블: `services`, `categories`, `feedbacks`, `users`, `promotions`, `service_entry_logs`, `report`
- 현재 API_SPEC에 있는 관련 API: `GET /services`, `POST /services`, `GET /services/{service_id}`, `PATCH /services/{service_id}`, `DELETE /services/{service_id}`, `GET /users/me/services`, `GET /services/{service_id}/feedback-trends`
- API_SPEC 보완 필요: 서비스 등록 조건 조회, URL 검증, AI 추천 API

## Step 계획

### Step 1 : 기능 계약 확인

**목표**

서비스 등록 기능에서 사용할 화면, API, 테이블 필드를 먼저 고정한다.

**구현 기능**

- 헤더의 `서비스` 버튼 클릭 시 서비스 페이지로 진입하는 라우트 정의
- 서비스 페이지 기본 레이아웃 정의
- 왼쪽 카테고리 메뉴와 오른쪽 서비스 목록 영역 구성
- `frontend` 브랜치에서 생성된 서비스 관련 화면과 컴포넌트 구조 확인
- `services`에 필요한 필드 정리: 서비스명, 한 줄 소개, 설명, URL, 카테고리, 썸네일, 상태, 작성자
- API_SPEC의 서비스 CRUD 명세 확인
- 서비스 등록 조건 조회 API 명세 초안 작성

**완료 조건**

- 서비스 목록, 상세, 등록 페이지의 라우트가 확정된다.
- 피그마 기반 서비스 화면 코드에서 수정해야 할 부분이 담당 범위 안에서 정리된다.
- `GET /services`, `POST /services`, `GET /services/{service_id}`, `PATCH /services/{service_id}`, `DELETE /services/{service_id}` 사용 기준이 정리된다.
- API_SPEC에 아직 없는 등록 조건, URL 검증, AI 추천 API가 보완 필요 항목으로 정리된다.
- 목업 데이터로 서비스 페이지 진입이 가능하다.

**엮이는 담당자**

- 김현진: 로그인 사용자와 관심 카테고리 정보
- 박재범: 홍보 노출 여부
- 박건일: 서비스 신고 대상 구조
- 김은경: 내 서비스 목록

**확인 방법**

- 헤더의 서비스 버튼에서 서비스 페이지로 이동한다.
- 목업 서비스 목록이 카테고리 영역과 함께 표시되는지 확인한다.

### Step 2 : 핵심 조회 흐름

**목표**

사용자가 서비스 페이지에서 전체 서비스와 카테고리별 서비스를 탐색할 수 있게 한다.

**구현 기능**

- 전체 서비스 조회
- 특정 서비스 조회
- 카테고리별 필터링
- 서비스 카드 표시: 썸네일 이미지, 서비스명, 한 줄 설명
- 한 페이지에 10개, 2열 5행 레이아웃
- 카테고리 목록 표시
- 홍보 중인 서비스 우선 노출을 위한 정렬 필드 협의

**완료 조건**

- 전체 탭과 카테고리별 목록이 정상적으로 바뀐다.
- `GET /services`와 `GET /services/{service_id}` 기준으로 목록/상세 조회가 동작한다.
- 서비스 카드 10개 기준 페이지 구성이 깨지지 않는다.
- 서비스 카드 클릭 시 상세 또는 등록된 서비스 확인 화면으로 이동한다.

**엮이는 담당자**

- 박재범: 홍보 중인 서비스 노출 우선순위
- 김현진: 서비스 상세의 피드백 목록 연결

**확인 방법**

- 전체 서비스 목록을 조회한다.
- 각 카테고리를 클릭해 목록이 바뀌는지 확인한다.
- 10개 초과 데이터에서 페이지 처리가 가능한지 확인한다.

### Step 3 : 핵심 생성/등록 흐름

**목표**

등록 조건을 만족한 사용자가 내 서비스를 등록할 수 있게 한다.

**구현 기능**

- 서비스 등록 페이지
- `POST /services` 기반 서비스 등록
- 입력 필드: 서비스명, 한 줄 소개, 서비스 설명, 서비스 URL, 서비스 카테고리
- 선택 입력: 서비스 썸네일
- 썸네일 제한: 1280x720, 3MB 이하, PNG/JPG
- 등록 성공 후 내 서비스 목록 또는 서비스 상세로 이동

**완료 조건**

- 필수값을 입력하면 서비스가 `services`에 저장된다.
- 등록 요청은 `POST /services` 명세를 따른다.
- 썸네일 조건을 벗어나면 등록 전에 차단된다.
- 등록된 서비스가 목록과 마이페이지 내 서비스 목록에서 조회 가능하다.

**엮이는 담당자**

- 김현진: 서비스 등록 가능 조건의 피드백 1건 작성 여부
- 김은경: 마이페이지 내 서비스 목록

**확인 방법**

- 조건을 만족한 계정으로 서비스를 등록한다.
- 등록 후 서비스 목록과 내 서비스 목록에 같은 서비스가 보이는지 확인한다.

### Step 4 : 수정/삭제 또는 상태 변경 흐름

**목표**

서비스의 공개 상태를 관리하고 신고 처리와 연결할 수 있게 한다.

**구현 기능**

- 서비스 상태값 정의: 정상 노출, 숨김, 차단 등
- `PATCH /services/{service_id}` 기반 서비스 수정
- `DELETE /services/{service_id}` 기반 서비스 삭제 또는 삭제 상태 처리
- 관리자 또는 신고 처리에 따른 서비스 블라인드 반영
- 비활성/차단 서비스가 목록과 홍보 노출에서 제외되도록 상태 기준 정의

**완료 조건**

- 블라인드 처리된 서비스는 일반 목록에 노출되지 않는다.
- 작성자는 `PATCH /services/{service_id}`로 본인 서비스를 수정할 수 있다.
- `DELETE /services/{service_id}` 호출 시 실제 삭제 또는 상태 변경 정책이 확정된다.
- 서비스 상태가 홍보 노출 판단에 사용된다.
- 상태값이 신고 처리 결과와 연결 가능하다.

**엮이는 담당자**

- 박건일: 서비스 신고 승인 시 블라인드 처리
- 박재범: 비활성/차단 서비스의 홍보 중단

**확인 방법**

- 서비스 상태를 숨김으로 바꾼 뒤 목록에서 제외되는지 확인한다.
- 숨김 서비스가 홍보 후보 또는 활성 홍보 노출에서 제외되는지 확인한다.

### Step 5 : 보상, 제한, 권한, 정책 검증

**목표**

첫 서비스 등록 조건과 URL 안전성 검사를 적용한다.

**구현 기능**

- 첫 서비스 등록 조건 체크리스트 모달
- 조건 1: 가입 24시간 경과
- 조건 2: 피드백 1개 작성
- 조건 달성 시마다 체크된 상태로 모달 표시
- 두 조건 모두 충족 시 `서비스 등록하러 가기` 버튼 노출
- Google Web Risk 기반 서비스 URL 검증

**완료 조건**

- 조건 미달 사용자는 등록 페이지 진입 또는 제출이 차단된다.
- 조건 충족 사용자는 체크리스트 모달에서 등록 페이지로 이동할 수 있다.
- 위험 URL은 저장되지 않는다.

**엮이는 담당자**

- 김현진: 가입 시각, 피드백 작성 여부
- 박건일: 위험 URL 신고 사유

**확인 방법**

- 신규 가입 직후 계정, 피드백만 작성한 계정, 24시간과 피드백 조건을 모두 만족한 계정으로 모달 상태를 확인한다.
- 위험 URL 목업 응답으로 등록 차단을 확인한다.

### Step 6 : 다른 담당 기능과의 1차 연결

**목표**

등록된 서비스가 피드백, 홍보, 신고, 마이페이지 기능과 연결되게 한다.

**구현 기능**

- 서비스 상세에서 피드백 목록과 작성 진입 연결
- 홍보 신청 시 본인 서비스 드롭다운에 등록 서비스 제공
- 서비스 신고 버튼과 신고 대상 정보 제공
- 내 서비스 목록 조회 응답에 필요한 필드 제공

**완료 조건**

- 서비스 상세에서 피드백 작성 화면으로 이동할 수 있다.
- 홍보 신청 모달에서 본인 서비스가 선택 가능하다.
- 서비스 신고 시 `target_type=service`, `target_id=service_id`로 접수 가능하다.
- 마이페이지에서 내 서비스가 조회된다.

**엮이는 담당자**

- 김현진: 피드백 작성
- 박재범: 홍보 신청
- 박건일: 신고
- 김은경: 마이페이지

**확인 방법**

- 하나의 등록 서비스로 피드백 작성, 홍보 신청, 신고, 마이페이지 조회를 모두 실행한다.

### Step 7 : 에러, 빈 상태, 예외 케이스 처리

**목표**

사용자가 등록 실패 사유와 빈 상태를 명확히 이해할 수 있게 한다.

**구현 기능**

- 서비스가 없는 카테고리 빈 상태
- 필수 입력값 누락 안내
- 썸네일 형식/용량/해상도 오류 안내
- URL 형식 오류, 접속 실패, Web Risk 실패 안내
- 비로그인 또는 조건 미달 안내

**완료 조건**

- 각 실패 케이스가 같은 오류 문구로 뭉개지지 않는다.
- 실패해도 입력 중이던 주요 값이 보존된다.
- 빈 목록 상태에서 레이아웃이 깨지지 않는다.

**엮이는 담당자**

- 김현진: 비로그인, 세션 만료
- 박건일: 신고로 인한 숨김 상태

**확인 방법**

- 필수값 누락, 잘못된 URL, 큰 썸네일, 조건 미달 계정으로 등록을 시도한다.
- 서비스 없는 카테고리를 조회한다.

### Step 8 : 관리자/운영 확인 또는 이력 조회

**목표**

운영자가 서비스 상태와 서비스 진입 데이터를 확인할 수 있는 최소 근거를 남긴다.

**구현 기능**

- 서비스 상세 진입 로그 저장
- 진입 로그 필드: 서비스, 회원, 세션, 진입 시각, 발생 페이지, 광고 종류
- 신고 처리와 홍보 성과 확인에 필요한 서비스 상태 유지

**완료 조건**

- 서비스 상세 진입 시 `service_entry_logs`에 기록이 남는다.
- 홍보 유입과 일반 유입을 구분할 수 있다.
- 숨김/차단 상태 변경 이력을 추적할 수 있는 데이터 기준이 정리된다.

**엮이는 담당자**

- 박재범: 홍보 노출 성과
- 박건일: 신고 처리 후 상태 확인

**확인 방법**

- 일반 목록과 홍보 영역에서 각각 서비스에 진입한 뒤 로그가 구분되는지 확인한다.

### Step 9 : 통합 테스트 보강

**목표**

핵심 순환에서 서비스 등록 구간이 끊기지 않는지 검증한다.

**구현 기능**

- 로그인 후 조건 확인
- 피드백 1건 작성 후 서비스 등록 가능 여부 갱신
- 서비스 등록 후 목록 노출
- 홍보 신청 후보 서비스로 노출
- 서비스 신고 접수 가능 여부 확인

**완료 조건**

- `develop`에서 피드백 작성 후 서비스 등록까지 진행된다.
- 등록된 서비스가 홍보, 신고, 마이페이지에서 동일한 식별자로 연결된다.

**엮이는 담당자**

- 김현진: 피드백과 로그인
- 박재범: 홍보
- 박건일: 신고
- 김은경: 마이페이지

**확인 방법**

- 통합 테스트 계정으로 로그인, 피드백 작성, 서비스 등록, 홍보 신청, 신고, 마이페이지 확인을 순서대로 실행한다.

### Step 10 : 배포 전 점검과 문서 정리

**목표**

서비스 등록 관련 명세와 확인 체크리스트를 배포 가능한 상태로 정리한다.

**구현 기능**

- API_SPEC의 서비스 CRUD 반영 여부 확인
