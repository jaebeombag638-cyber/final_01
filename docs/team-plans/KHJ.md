# 김현진 1차 배포 계획: 피드백, 소셜 로그인, 최초 관심분야 입력

## 담당 범위

- 구글 소셜 로그인 모달과 OAuth 흐름
- 최초 로그인 시 닉네임 설정
- 관심 카테고리 0~3개 선택 또는 나중에 선택
- 서비스 등록 조건 안내 모달
- 피드백 작성, 수정, 삭제
- 피드백 작성 제한과 품질 검사
- 피드백 보상 지급 이벤트 연결

## 작업 브랜치

- 기본 브랜치: `feature/login`, `feature/onboarding-interests`, `feature/feedback`
- 필요 시 세부 브랜치: `feature/feedback-quality-check`, `feature/feedback-delete-policy`

## 프론트엔드 작업 전제

- 피그마 MCP로 생성한 프론트 기초 코드는 `frontend` 브랜치에서 먼저 구현되고, 통합 관리자가 `develop`에 병합한다.
- 김현진 작업은 `frontend`가 병합된 `develop`에서 `feature/login`, `feature/onboarding-interests`, `feature/feedback` 브랜치를 만든 뒤 시작한다.
- 로그인 모달, 온보딩 화면, 피드백 작성/수정/삭제 화면은 피그마 기반 기초 코드를 최대한 유지한다.
- 피그마 의도와 다르게 생성된 인증/온보딩/피드백 화면 구조나 컴포넌트만 담당 범위 안에서 보정한다.
- 기능 연결 시 목업 로그인 상태와 목업 피드백 데이터는 실제 인증/온보딩/피드백 API 호출로 교체한다.

## 주요 계약

- 주요 테이블: `users`, `user_interests`, `categories`, `services`, `feedbacks`, `point_logs`
- 현재 API_SPEC 관련 API: `GET /auth/google`, `POST /auth/google/callback`, `POST /auth/logout`, `POST /onboarding`, `GET /users/nickname/check`, `GET /interest_categories`, 피드백 CRUD API, `GET /users/me/feedback-quota`
- 정책 상수: 관심 카테고리 최대 3개, 일일 피드백 3건, 서비스 1개당 사용자 1피드백, 피드백 삭제 30일 제한

## Step 계획

### Step 1 : 기능 계약 확인

**목표**

로그인, 온보딩, 피드백 기능의 요청/응답과 화면 흐름을 고정한다.

**구현 기능**

- 홈페이지 우측 상단 로그인 버튼과 로그인 모달 정의
- `frontend` 브랜치에서 생성된 로그인 모달, 온보딩, 피드백 화면 구조 확인
- 구글 아이콘과 구글 계정 로그인 버튼 구성
- OAuth URL 조회와 콜백 처리 흐름 정의
- 최초 로그인 사용자에게 onboarding token을 발급하는 흐름 정의
- 피드백 작성/수정/삭제 API 응답 형식 정의

**완료 조건**

- 기존 회원은 세션을 받고 최초 회원은 온보딩으로 이동한다.
- 피그마 기반 인증/온보딩/피드백 화면 코드에서 수정해야 할 부분이 담당 범위 안에서 정리된다.
- 로그인 후 원래 보던 창으로 돌아갈 수 있는 redirect 기준이 정리된다.
- 피드백 API의 권한 실패, 품질 실패, 제한 실패 응답이 구분된다.

**엮이는 담당자**

- 박재범: 최초 가입 보상
- 조현정: 서비스 등록 조건
- 김은경: 마이페이지 프로필

**확인 방법**

- 로그인 모달을 열고 구글 OAuth 진입 URL을 확인한다.
- 목업 콜백으로 기존 회원/신규 회원 응답 차이를 확인한다.

### Step 2 : 핵심 조회 흐름

**목표**

사용자가 자신의 로그인 상태, 관심 카테고리, 피드백 작성 가능 상태를 조회할 수 있게 한다.

**구현 기능**

- 관심 카테고리 목록 조회
- 닉네임 중복 확인
- 내 피드백 목록 조회
- 서비스별 피드백 목록 조회
- 오늘 피드백 작성 가능 횟수 조회
- 피드백 단건 조회

**완료 조건**

- 관심 카테고리 10개와 `나중에 선택할래요` 선택지가 온보딩에 표시된다.
- 피드백 작성 가능 횟수가 일일 3건 기준으로 계산된다.
- 서비스 상세에서 피드백 목록이 조회된다.

**엮이는 담당자**

- 조현정: 서비스 상세 화면
- 김은경: 내 피드백 목록

**확인 방법**

- 온보딩 화면에서 카테고리 목록을 확인한다.
- 서비스 상세에서 피드백 목록과 작성 가능 횟수를 조회한다.

### Step 3 : 핵심 생성/등록 흐름

**목표**

신규 사용자가 가입을 완료하고 로그인 사용자가 피드백을 작성할 수 있게 한다.

**구현 기능**

- 최초 로그인 시 닉네임 필수 설정
- 관심 카테고리 0~3개 선택
- `나중에 선택할래요` 선택 시 관심 카테고리 없이 온보딩 완료
- 정식 세션 발급
- 로그인한 사용자만 피드백 작성 가능
- 미로그인 상태에서는 어떤 글도 작성 불가

**완료 조건**

- 닉네임 중복이면 온보딩 완료가 차단된다.
- 관심 카테고리 4개 이상 선택이 차단된다.
- 온보딩 완료 후 정식 세션으로 서비스 탐색과 피드백 작성이 가능하다.

**엮이는 담당자**

- 박재범: 신규 가입 보상
- 박건일: 게시글/댓글 작성 권한
- 조현정: 서비스 등록 조건 안내

**확인 방법**

- 신규 계정으로 닉네임 설정, 관심 카테고리 선택, 온보딩 완료를 진행한다.
- 미로그인 상태에서 피드백/게시글/댓글 작성을 시도해 차단되는지 확인한다.

### Step 4 : 수정/삭제 또는 상태 변경 흐름

**목표**

피드백 수정과 삭제 정책을 구현한다.

**구현 기능**

- 피드백 작성자만 수정 가능
- 수정 시에도 품질 검사 재실행
- 피드백 삭제 클릭 시 모달 표시
- 작성 30일 미만 삭제 차단 모달
- 작성 30일 경과 삭제 확인 모달
- 삭제해도 지급된 빡낌은 회수되지 않는다는 안내 표시

**완료 조건**

- 본인이 아닌 피드백은 수정/삭제할 수 없다.
- 30일 미만 피드백 삭제가 차단된다.
- 30일 경과 피드백은 확인 후 삭제된다.

**엮이는 담당자**

- 박재범: 삭제 시 포인트 미회수 정책
- 김은경: 마이페이지 내 피드백 표시

**확인 방법**

- 30일 미만/이상 피드백 데이터를 각각 삭제 시도한다.
- 피드백 수정 후 품질 검사 결과가 다시 반영되는지 확인한다.

### Step 5 : 보상, 제한, 권한, 정책 검증

**목표**

피드백 작성 제한과 품질 검사를 적용한다.

**구현 기능**

- 로그인 사용자만 피드백 작성 가능
- 본인 서비스에는 피드백 작성 불가
- 하루 피드백 3건 작성 제한
- 서비스 1건당 사용자 1개 피드백 제한
- 가입 24시간 지난 사용자의 첫 피드백 등록 시 서비스 등록 조건 충족
- 로직 검사: 공백, 자음, 모음, 기호 제외 100자 이상
- LLM 품질 검사
- 승인된 피드백만 등록

**완료 조건**

- 본인 서비스, 중복 작성, 일일 제한 초과가 차단된다.
- 100자 미만 실질 내용은 품질 검사 전에 차단된다.
- LLM 품질 승인 결과만 저장된다.

**엮이는 담당자**

- 조현정: 서비스 등록 조건
- 박재범: 피드백 보상

**확인 방법**

- 본인 서비스, 타인 서비스, 중복 피드백, 일일 4번째 피드백을 각각 시도한다.
- 100자 미만 내용과 정상 내용을 각각 작성한다.

### Step 6 : 다른 담당 기능과의 1차 연결

**목표**

피드백 성공 이벤트가 서비스 등록 조건과 포인트 지급에 연결되게 한다.

**구현 기능**

- 피드백 1건당 200빡낌 지급 이벤트 전달
- 서비스 최초 피드백이면 300빡낌 지급 이벤트 전달
- 가입 24시간 경과 + 피드백 1건 작성 시 최초 1회 축하 모달 표시
- 축하 모달의 `내 서비스 등록하러 가기` 버튼 연결

**완료 조건**

- 승인된 피드백 작성 후 포인트 지급이 호출된다.
- 서비스 첫 피드백 여부가 구분된다.
- 서비스 등록 조건 충족 모달이 최초 1회만 표시된다.

**엮이는 담당자**

- 박재범: 포인트 지급
- 조현정: 서비스 등록 페이지 이동

**확인 방법**

- 타인 서비스에 첫 피드백을 작성하고 포인트 지급 로그와 서비스 등록 모달을 확인한다.

### Step 7 : 에러, 빈 상태, 예외 케이스 처리

**목표**

로그인/온보딩/피드백 실패 사유를 사용자에게 명확히 안내한다.

**구현 기능**

- OAuth 실패 안내
- 세션 만료 안내
- 닉네임 중복 안내
- 관심 카테고리 3개 초과 안내
- 피드백 제한 초과 안내
- 품질 검사 실패 안내
- 삭제 불가 안내

**완료 조건**

- 실패 케이스별 메시지가 구분된다.
- 실패 후 사용자가 다음 행동을 알 수 있다.
- 입력 중이던 피드백 내용은 가능한 한 유지된다.

**엮이는 담당자**

- 박건일: 게시글/댓글 작성 권한 공통 처리
- 김은경: 프로필 수정 닉네임 중복

**확인 방법**

- 각 실패 케이스를 수동으로 발생시키고 모달/토스트/응답 메시지를 확인한다.

### Step 8 : 관리자/운영 확인 또는 이력 조회

**목표**

피드백 품질 검사와 제한 판단에 필요한 데이터를 남긴다.

**구현 기능**

- 피드백 글자수 저장
- 품질 검사 결과 저장
- 작성/수정 시각 저장
- 사용자별 일일 작성 횟수 계산 근거 유지
- 서비스별 최초 피드백 판단 근거 유지

**완료 조건**

- 포인트 지급과 등록 조건 판단에 필요한 피드백 상태가 추적 가능하다.
- 품질 반려 사유가 운영 확인 가능하다.

**엮이는 담당자**

- 박재범: 포인트 지급 근거
- 박건일: 신고된 피드백 확인

**확인 방법**

- 승인/반려 피드백을 만든 뒤 저장된 글자수, 품질 상태, 작성 시각을 확인한다.

### Step 9 : 통합 테스트 보강

**목표**

가입부터 피드백 작성, 포인트 지급, 서비스 등록 조건 충족까지 검증한다.

**구현 기능**

- 신규 가입 시나리오
- 관심 카테고리 선택/건너뛰기
- 피드백 작성
- 포인트 지급 이벤트 확인
- 서비스 등록 가능 모달 확인

**완료 조건**

- `develop`에서 신규 사용자가 온보딩 후 피드백을 작성할 수 있다.
- 피드백 작성 후 서비스 등록 조건과 포인트 잔액이 갱신된다.

**엮이는 담당자**

- 조현정: 서비스 등록
- 박재범: 포인트

**확인 방법**

- 신규 계정으로 로그인, 온보딩, 피드백 작성, 포인트 잔액 확인, 서비스 등록 모달 확인을 순서대로 실행한다.

### Step 10 : 배포 전 점검과 문서 정리

**목표**

로그인, 온보딩, 피드백 정책을 배포 전 확인 가능한 문서로 정리한다.

**구현 기능**

- 로그인 성공/실패 확인법
- 온보딩 확인법
- 피드백 제한 정책 정리
- 품질 검사 기준 정리
- 포인트 지급 이벤트 조건 정리
- 삭제 정책 모달 문구 정리

**완료 조건**

- QA가 문서만 보고 로그인/피드백 핵심 시나리오를 재현할 수 있다.
- Function_def의 로그인과 피드백 요구사항이 누락 없이 반영된다.

**엮이는 담당자**

- 통합 관리자

**확인 방법**

- 배포 전 체크리스트로 OAuth, 온보딩, 피드백 작성/수정/삭제, 제한, 보상을 재확인한다.
