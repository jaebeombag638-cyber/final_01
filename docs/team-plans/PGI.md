# 박건일 1차 배포 계획: 커뮤니티 기능 전체, 신고

## 담당 범위

- 커뮤니티 게시판 기능
- 게시글 목록, 상세, 작성, 수정, 삭제
- 게시글 필터링 탭과 정렬
- 욕설 사전 기반 게시글 필터링
- 서비스, 피드백, 커뮤니티 글, 커뮤니티 댓글 신고
- 신고 모달
- 관리자 신고 내역 확인과 블라인드 처리

## 작업 브랜치

- 기본 브랜치: `feature/posts`, `feature/reports`
- 필요 시 세부 브랜치: `feature/post-filtering`, `feature/report-admin`

## 프론트엔드 작업 전제

- 피그마 MCP로 생성한 프론트 기초 코드는 `frontend` 브랜치에서 먼저 구현되고, 통합 관리자가 `develop`에 병합한다.
- 박건일 작업은 `frontend`가 병합된 `develop`에서 `feature/posts`, `feature/reports` 브랜치를 만든 뒤 시작한다.
- 커뮤니티 목록/상세/작성 화면, 신고 모달, 관리자 신고 화면은 피그마 기반 기초 코드를 최대한 유지한다.
- 피그마 의도와 다르게 생성된 커뮤니티/신고 화면 구조나 컴포넌트만 담당 범위 안에서 보정한다.
- 기능 연결 시 목업 게시글과 목업 신고 데이터는 실제 커뮤니티/신고 API 호출로 교체한다.

## 주요 계약

- 주요 테이블: `community_posts`, `community_comments`, `report`, `services`, `feedbacks`, `users`
- 현재 API_SPEC 관련 API: `GET /posts?type={post_type}&sort={sort_type}`, `GET /posts/{post_id}`, `POST /posts`, `PUT /posts/{post_id}`, `DELETE /posts/{post_id}`, `GET /reports`, `POST /reports`, `PATCH /reports/{report_id}`
- 신고 대상: 서비스, 피드백, 커뮤니티 글, 커뮤니티 댓글
- 신고 유형: 불법/음란물 광고, 도배 및 스팸, 욕설 및 비방, 표절 및 도용, 기타 및 상세 사유

## Step 계획

### Step 1 : 기능 계약 확인

**목표**

커뮤니티 게시글과 신고 기능의 화면, API, 상태값을 고정한다.

**구현 기능**

- 게시판 탭 정의: 전체, 자유게시판, 팀 구하기, 건의하기
- `frontend` 브랜치에서 생성된 커뮤니티 화면, 신고 모달, 관리자 신고 화면 구조 확인
- 게시글 정렬 기준 정의: 조회수, 최신 순
- 게시글 카드 필드 정의: 제목, 내용, 닉네임, 생성일자, 카테고리, 조회수
- 신고 대상 타입 정의: 서비스, 피드백, 커뮤니티, 커뮤니티 댓글
- 신고 처리 상태 정의: 승인됨, 거부됨

**완료 조건**

- 게시글과 신고 API 요청/응답 필드가 정리된다.
- 피그마 기반 커뮤니티/신고 화면 코드에서 수정해야 할 부분이 담당 범위 안에서 정리된다.
- 신고 모달에 표시할 신고 유형 5개가 확정된다.
- 관리자 페이지에서 사용할 신고 목록 필드가 정리된다.

**엮이는 담당자**

- 김현진: 로그인 사용자와 피드백 신고
- 조현정: 서비스 신고와 서비스 블라인드
- 김은경: 댓글 신고

**확인 방법**

- 목업 데이터로 게시글 목록, 게시글 상세, 신고 모달, 관리자 신고 목록을 표시한다.

### Step 2 : 핵심 조회 흐름

**목표**

사용자가 커뮤니티 게시글을 탐색하고 관리자가 신고 내역을 조회할 수 있게 한다.

**구현 기능**

- 전체/자유게시판/팀 구하기/건의하기 탭 조회
- 조회수 정렬과 최신 순 정렬
- 게시글 카드 목록 조회
- 게시글 상세 조회
- 관리자 신고 목록 조회
- 관리자 신고 목록 탭: 전체, 서비스, 피드백, 커뮤니티, 커뮤니티 댓글

**완료 조건**

- 게시판 탭을 바꾸면 해당 카테고리 글만 보인다.
- 정렬 기준을 바꾸면 조회수/최신 순으로 목록이 바뀐다.
- 신고 내역이 신고한 순서대로 조회된다.

**엮이는 담당자**

- 김은경: 게시글 상세 하단 댓글 목록
- 조현정: 서비스 신고 내역 탭
- 김현진: 피드백 신고 내역 탭

**확인 방법**

- 각 게시판 탭과 정렬 옵션을 클릭한다.
- 관리자 계정으로 신고 목록의 대상 타입별 탭을 확인한다.

### Step 3 : 핵심 생성/등록 흐름

**목표**

로그인 사용자가 게시글을 작성하고 사용자가 콘텐츠를 신고할 수 있게 한다.

**구현 기능**

- 게시글 작성
- 게시글 카테고리 선택
- 건의하기 글 등록
- 신고 버튼 클릭 시 신고 모달 표시
- 신고 유형 선택 또는 기타 상세 사유 입력
- 신고 접수

**완료 조건**

- 로그인 사용자가 게시글을 작성하면 목록에 노출된다.
- 건의하기 글은 등록 직후 모든 사용자가 볼 수 있다.
- 신고 모달에서 신고 유형을 선택하거나 기타 사유를 입력해 접수할 수 있다.

**엮이는 담당자**

- 김현진: 미로그인 작성 차단
- 조현정: 서비스 신고 버튼
- 김은경: 댓글 신고 버튼

**확인 방법**

- 로그인 상태에서 게시글을 작성한다.
- 서비스, 피드백, 게시글, 댓글 신고를 각각 접수한다.

### Step 4 : 수정/삭제 또는 상태 변경 흐름

**목표**

작성자 게시글 관리와 관리자 신고 처리를 구현한다.

**구현 기능**

- 작성자만 게시글 수정 가능
- 작성자만 게시글 삭제 가능
- 건의하기 글을 관리자가 비밀글 또는 블라인드 처리 가능
- 신고 승인 시 대상 게시물 블라인드 처리
- 신고 거부 시 대상 유지

**완료 조건**

- 게시글 작성자 외 사용자는 수정/삭제 버튼을 사용할 수 없다.
- 신고 승인 처리 시 `target_id`의 게시글 또는 대상 콘텐츠가 블라인드된다.
- 신고 거부 처리 시 콘텐츠 노출 상태가 유지된다.

**엮이는 담당자**

- 조현정: 서비스 블라인드 처리
- 김은경: 댓글 블라인드 처리
- 김현진: 피드백 신고 처리 대상

**확인 방법**

- 타인 게시글 수정/삭제를 시도한다.
- 관리자 신고 승인 후 대상 콘텐츠가 블라인드되는지 확인한다.

### Step 5 : 보상, 제한, 권한, 정책 검증

**목표**

커뮤니티 작성 권한과 텍스트 필터링을 적용한다.

**구현 기능**

- 미로그인 사용자의 게시글 작성 차단
- 미로그인 사용자의 신고 차단 여부 확정
- 미리 정의된 욕설 사전 기반 텍스트 단순 비교
- 금칙어 포함 시 작성/수정 차단
- 관리자 페이지 접근 권한 확인

**완료 조건**

- 욕설 사전에 포함된 단어가 있으면 게시글 등록/수정이 차단된다.
- 일반 사용자는 관리자 신고 페이지에 접근할 수 없다.
- 작성 권한 실패와 필터링 실패가 구분된다.

**엮이는 담당자**

- 김현진: 로그인/세션 권한
- 김은경: 댓글 필터링 정책 공유

**확인 방법**

- 금칙어가 포함된 게시글을 작성한다.
- 일반 사용자로 관리자 신고 API에 접근한다.

### Step 6 : 다른 담당 기능과의 1차 연결

**목표**

신고 기능이 서비스, 피드백, 게시글, 댓글에 공통으로 연결되게 한다.

**구현 기능**

- 서비스 신고 연결
- 피드백 신고 연결
- 커뮤니티 글 신고 연결
- 커뮤니티 댓글 신고 연결
- 신고 대상 타입과 대상 아이디 저장

**완료 조건**

- 네 가지 대상의 신고가 모두 `report` 테이블에 같은 구조로 저장된다.
- 관리자 신고 목록에서 대상 타입별로 필터링할 수 있다.
- 신고 승인 결과가 각 담당 기능의 숨김/블라인드 상태로 전달된다.

**엮이는 담당자**

- 조현정: 서비스 신고
- 김현진: 피드백 신고
- 김은경: 댓글 신고

**확인 방법**

- 대상별 신고를 1건씩 접수하고 관리자 탭에서 분류되어 보이는지 확인한다.

### Step 7 : 에러, 빈 상태, 예외 케이스 처리

**목표**

커뮤니티와 신고의 빈 상태 및 예외 상황을 처리한다.

**구현 기능**

- 게시글 없는 게시판 빈 상태
- 신고 내역 없는 관리자 탭 빈 상태
- 삭제 또는 블라인드 게시글 접근 안내
- 존재하지 않는 신고 대상 처리
- 신고 사유 미선택 처리
- 기타 신고 시 상세 사유 미입력 처리

**완료 조건**

- 빈 목록과 오류 상태에서 화면이 깨지지 않는다.
- 신고 유형 미선택 또는 기타 사유 누락 시 접수가 차단된다.
- 블라인드된 게시글은 일반 사용자에게 내용이 노출되지 않는다.

**엮이는 담당자**

- 김은경: 블라인드 댓글 표시
- 조현정: 블라인드 서비스 표시

**확인 방법**

- 빈 게시판, 빈 신고 목록, 삭제 게시글, 신고 사유 누락 케이스를 확인한다.

### Step 8 : 관리자/운영 확인 또는 이력 조회

**목표**

관리자가 신고 내역을 검토하고 정책 위반 대상을 처리할 수 있게 한다.

**구현 기능**

- 신고 내역을 신고한 순서대로 표시
- 신고 유형, 신고 사유, 처리 상태 표시
- 대상 타입별 탭 필터
- 승인/거부 처리 버튼
- 승인 처리 시 대상 블라인드 버튼 또는 자동 블라인드 처리

**완료 조건**

- 관리자는 전체 신고 내역과 대상별 신고 내역을 모두 확인한다.
- 신고 승인/거부 상태가 저장된다.
- 승인된 신고의 대상 콘텐츠가 블라인드된다.

**엮이는 담당자**

- 조현정: 서비스 블라인드
- 김현진: 피드백 처리 상태
- 김은경: 댓글 블라인드

**확인 방법**

- 관리자 계정으로 신고 1건을 승인하고 1건을 거부한다.
- 승인 대상이 일반 사용자 화면에서 숨겨지는지 확인한다.

### Step 9 : 통합 테스트 보강

**목표**

커뮤니티와 신고가 다른 기능과 함께 통합 동작하는지 검증한다.

**구현 기능**

- 게시글 작성 후 댓글 작성 연결 확인
- 게시글 신고 접수와 관리자 처리
- 서비스/피드백/댓글 신고 접수와 관리자 탭 분류
- 블라인드 처리 후 각 화면 노출 확인

**완료 조건**

- `develop`에서 네 가지 신고 대상이 모두 접수되고 관리자 처리된다.
- 블라인드 결과가 서비스, 피드백, 게시글, 댓글 화면에 반영된다.

**엮이는 담당자**

- 조현정, 김현진, 김은경

**확인 방법**

- 통합 테스트 계정으로 게시글 작성, 댓글 작성, 각 대상 신고, 관리자 승인, 화면 재조회까지 진행한다.

### Step 10 : 배포 전 점검과 문서 정리

**목표**

커뮤니티와 신고 운영 기준을 배포 전 확인 가능한 문서로 정리한다.

**구현 기능**

- 게시판 타입과 정렬 기준 정리
- 욕설 사전 기반 필터링 기준 정리
- 신고 유형 5개 문구 정리
- 신고 승인/거부 처리 절차 정리
- 블라인드 처리 확인법 정리

**완료 조건**

- QA가 게시글 CRUD와 신고 처리 전체를 재현할 수 있다.
- Function_def의 커뮤니티/신고 요구사항이 누락 없이 반영된다.

**엮이는 담당자**

- 통합 관리자

**확인 방법**

- 배포 전 체크리스트로 게시글 조회/작성/수정/삭제, 신고 접수, 관리자 처리를 재확인한다.
